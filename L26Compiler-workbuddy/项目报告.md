# 2026编译原理大作业项目报告

# 1 项目概述

## 1.1 作业需求说明
根据《2026编译实践大作业要求》：
1. 实现扩展L26语言编译器，支持`int`整型、`bool`布尔、`set`集合三种数据类型；
2. set集合元素上限200；
3. 编译器输出标准类P-Code三字段指令，配套虚拟机解释运行；
4. 语言语法包含块、变量声明、赋值、if-else、while循环、读写语句、集合专用语法；
5. 集合功能：集合字面量`{}`、`union`并集、`inter`交集、`add/remove`增删、`in`成员判断、`isempty`空集判断；
6. 支持多层嵌套作用域、变量遮蔽、完整静态语义检查；
7. 界面展示P-Code清单、内存管理、单步运行（加分项）；
8. 集合相等`==`/不等`!=`判定（加分项）。

## 1.2 开发方案选型
本项目选择**第三组 Coco/R C++实现**：
- Coco/R从ATG文法文件自动生成词法扫描器与LL(1)递归下降语法解析器；
- 在ATG文法中嵌入语义动作，同步完成符号表管理、类型检查、P-Code代码生成；
- 采用标准PL/0三字段P-Code格式，设计集合扩展OPR子码；
- 前端提供Web可视化界面，通过Node.js桥接调用C++后端。

## 1.3 整体架构

```
L26源程序(.l26)
    │
    ▼
Coco/R Scanner（自动生成）→ Token流
    │
    ▼
Coco/R Parser（ATG + 语义动作）
    ├── SymTab：嵌套作用域管理、类型检查
    └── PCodeGen：生成标准三字段P-Code指令
    │
    ▼
P-Code指令序列
    │
    ├──▶ C++ VM（VM.h）：命令行解释执行
    │
    └──▶ JSON API → Node.js桥接 → 前端JS VM：可视化单步调试
```

**两层运行模式：**

- **普通模式**：`l26compiler.exe <file>` 编译并可选执行虚拟机
- **JSON模式**：`l26compiler.exe <file> --json` 输出结构化JSON供前端使用
- **前端模式**：`node server.js` 启动HTTP桥接，浏览器访问`localhost:8765`使用可视化界面

# 2 L26语言完整EBNF文法定义

## 2.1 EBNF元符号
- `::=` 定义；`|` 或；`{}` 0次及以上重复；`[]` 可选；双引号为终结符

## 2.2 完整EBNF文法

```ebnf
<program>        ::= <block>
<block>          ::= "{" <decls> <stmts> "}"
<decls>          ::= { <decl> }
<decl>           ::= <type> ID ";"
<type>           ::= "int" | "bool" | "set"

<stmts>          ::= { <stmt> }
<stmt>           ::= <assign_stmt> | <if_stmt> | <while_stmt>
                   | <io_stmt> | <block> | <set_op_stmt>

<assign_stmt>    ::= ID "=" <expr> ";"
<if_stmt>        ::= "if" "(" <bexpr> ")" <stmt> [ "else" <stmt> ]
<while_stmt>     ::= "while" "(" <bexpr> ")" <stmt>

<io_stmt>        ::= "write" <expr> ";" | "read" ID ";"
<set_op_stmt>    ::= "add" ID <aexpr> ";" | "remove" ID <aexpr> ";"

<expr>           ::= <bexpr> | <aexpr> | <set_expr>

/* 算术表达式 */
<aexpr>          ::= <aterm> { ("+" | "-") <aterm> }
<aterm>          ::= <afactor> { ("*" | "/") <afactor> }
<afactor>        ::= ID | NUM | "(" <aexpr> ")"

/* 逻辑表达式 */
<bexpr>          ::= <bterm> { "||" <bterm> }
<bterm>          ::= <bfactor> { "&&" <bfactor> }
<bfactor>        ::= "true" | "false" | "!" <bfactor>
                   | "(" <bexpr> ")" | <rel> | <set_test>

/* 关系运算 */
<rel>            ::= <aexpr> ("<" | "<=" | ">" | ">=" | "==" | "!=") <aexpr>

/* 集合表达式 */
<set_expr>       ::= "{" [ <aexpr> { "," <aexpr> } ] "}"
                   | ID "union" ID | ID "inter" ID

/* 集合测试 */
<set_test>       ::= <aexpr> "in" ID | "isempty" "(" ID ")"
```

## 2.3 关键字与运算符
- 关键字：int、bool、set、if、else、while、read、write、add、remove、union、inter、in、isempty、true、false
- 算术运算符：`+ - * /`
- 关系运算符：`< <= > >= == !=`
- 逻辑运算符：`! && ||`
- 集合运算符：`union inter in`

## 3.2 后端核心模块（Compiler/）

### 3.2.1 Coco/R文法文件（L26.atg）

L26.atg是整个编译器的核心，Coco/R据此自动生成Scanner和Parser。文件结构分四节：

**CHARACTERS节**：定义字符集（letter = `A..Z a..z _`，digit = `0..9`）。

**TOKENS节**：定义Token识别规则。`ident = letter {letter | digit}` 识别标识符，`number = digit {digit}` 识别整数。关键字由Coco/R的KeywordMap在扫描阶段自动识别，无需在TOKENS逐一列举。

**COMMENTS/IGNORE节**：支持`//`单行注释和`/* */`嵌套块注释，空白字符被忽略。

**PRODUCTIONS节**：每个产生式对应一个Parser方法，花括号`(. ... .)`内嵌C++语义动作，在语法匹配时同步执行。关键产生式及语义动作：

| 产生式 | 语义动作 |
|--------|----------|
| `Block` | `EnterScope()` → 发射INT(占位) → `Decls`统计变量数 → 回填INT.a = cnt+3 → `Stmts` → `LeaveScope()` |
| `Decl` | 调用`Declare()`检查重复声明，分配偏移 |
| `AssignStmt` | `Lookup()`查变量 → `Expr()`算类型 → 类型匹配检查 → `EmitSTO()`存入 |
| `IfStmt` | 发射JPC占位 → 解析then体 → 发射JMP占位 → `PatchJump(JPC)` → [解析else体] → `PatchJump(JMP)` |
| `WhileStmt` | 记录labelTop → 发射JPC占位 → 解析循环体 → 发射JMP回跳 → `PatchJump(JPC)` |
| `Expr` | 用`IF(IsSetExprStart())`/`IF(IsAExprStart())`谓词消歧，后置`[...]`检测relop/in/union/inter |
| `SetExpr` | 计数元素 → `EmitSetMake(cnt)`；union/inter侧做set类型校验 |

### 3.2.2 符号表（SymTab.h）

实现栈式多级作用域，纯头文件inline实现。

**数据结构**：
```cpp
struct SymEntry {
    string name;     // 变量名
    string typName;  // "int" / "bool" / "set"
    int    level;    // 定义所在嵌套层（0为最外层）
    int    offset;   // 当前块内偏移
};
```
符号表本体为 `vector<map<string, SymEntry>> scopes`，每一层作用域对应一个map。辅助变量 `curLevel` 记录当前嵌套深度，`offsetStack` 记录每层已分配偏移数。

**核心方法**：
- `EnterScope()`：curLevel++，压入空map和初始偏移0
- `LeaveScope()`：curLevel--，弹出栈顶map和偏移
- `Declare(name, typ)`：在scopes[curLevel]中插入，同层重名返回false
- `Lookup(name)`：从curLevel向下遍历scopes[0]，返回最近绑定，实现变量遮蔽
- `CurrentOffset()`：返回当前层已分配偏移数

**偏移**：变量偏移从0开始计数，代码生成时加3（`offset+3`），因为活动记录前3槽固定为SL/DL/RA联系单元。

### 3.2.3 P-Code代码生成器（PCode.h）

包含三部分：

**指令结构体**： 

```cpp
struct PInstr {
    fct f;  // 操作码（F_LIT/F_OPR/F_LOD/F_STO/F_CAL/F_INT/F_JMP/F_JPC）
    int l;  // 层次差（静态链向上层数，无层次时为0）
    int a;  // 操作数（常量值/变量偏移/跳转地址/OPR子码）
};
```
严格遵循三元组(f, l, a)格式。前缀`F_`为规避Windows SDK中CAL/INT宏冲突。

**操作码枚举**：8种基础操作码覆盖全部PL/0标准指令。

**OPR子码枚举**：标准子码（0-20）保持与PL/0一致，新增集合扩展子码（201-211）。

**代码生成器类PCodeGen**：

- `Emit(f, l, a)`：底层发射，返回指令序号
- 封装函数：`EmitLIT(val)`、`EmitLOD(level, off)`、`EmitSTO(level, off)`、`EmitINT(size)`、`EmitRET()`、`EmitJMP(addr)`、`EmitJPC(addr)`、`EmitBinOp(op)`、`EmitRelOp(op)`、`EmitBoolOp(kind)`、`EmitBoolNot()`、`EmitWRITE(typ)`、`EmitWRITELN()`、`EmitREAD()`
- 集合发射：`EmitSetMake(cnt)`（cnt存l字段）、`EmitSetUnion()`、`EmitSetInter()`、`EmitSetIn()`、`EmitSetIsEmpty()`、`EmitSetAdd()`、`EmitSetRemove()`、`EmitSetEQ()`、`EmitSetNE()`
- `PatchJump(idx)`：将idx处指令的a字段改为当前地址，解决if/while前向跳转
- `PrintListing()`：格式化输出全部P-Code汇编清单

**关键设计**：`EmitSetMake(cnt)`将元素数存入l字段（`Emit(F_OPR, cnt, OPR_SET_MK)`），a字段保留OPR子码值201，确保VM可通过`case OPR_SET_MK`正确匹配。

### 3.2.4 虚拟机（VM.h）

**寄存器**：
- `s[2048]`：整型操作数栈
- `p`：程序计数器（当前指令序号）
- `b`：基址寄存器（当前活动记录起始地址）
- `t`：栈顶寄存器

**静态链寻址**：

```cpp
int Base(int l, int cur_b) {
    while (l > 0) { cur_b = s[cur_b]; l--; }
    return cur_b;
}
```
沿`s[b]`（即SL）向上跳l层，获取目标层基址。LOD/STO指令的偏移相对该基址计算。

**活动记录布局**：
```
s[b+0] = SL  // 指向上一层活动记录的b
s[b+1] = DL  // 调用方b（主程序为0）
s[b+2] = RA  // 返回地址（主程序为0）
s[b+3..b+2+n] = n个局部变量
```
INT指令分配`a`个栈槽（`t += a`），其中`a = 局部变量数 + 3`。RET指令销毁帧：`t = b-1; p = s[b+2]; b = s[b+1]`。

**集合扩展存储**：`static set<int> g_setPool[2048]`，栈中仅存集合池下标。VM通过`g_setPool[t]`访问集合内容，不破坏整型栈结构。集合200上限`MAX_SET_SIZE=200`在SET_MK（构造时检查）、SET_UNION（并集后检查）、SET_ADD（添加前检查）三处强制。

**执行流程**：

1. `Run()`：循环调用`ExecInstr()`直至halted或PC越界
2. `Step()`：单步执行一条指令
3. `ExecInstr()`：switch分发8种基础指令
4. `ExecOPR()`：switch分发30多种OPR子码，涵盖算术、关系、逻辑、IO、全部集合操作

### 3.2.5 主程序入口（main.cpp）

支持两种运行模式：

**普通模式**（`l26compiler.exe <file>`）：

1. `system("chcp 65001")` 设置UTF-8控制台
2. 创建Scanner读取源文件
3. 创建Parser执行语法/语义分析
4. 有错误则打印计数并退出
5. 无错误则打印P-Code清单，询问是否执行VM
6. 初始化VM加载P-Code，执行并输出结果

**JSON模式**（`l26compiler.exe <file> --json`）：
1. 第一遍：单独创建Scanner，循环调用`Scan()`收集全部Token（类型、值、行号、列号）
2. 第二遍：新建Scanner+Parser，设置`errors->silent=true`抑制wprintf输出，调用`Parse()`收集P-Code
3. 收集错误：从`errors->errorMsgs`（`vector<ErrorEntry>`）读取，wchar_t→UTF-8正确编码中文字符
4. P-Code附带估算行号：按指令序号比例映射到token流位置
5. 输出一行完整JSON（tokens数组 + pcode数组 + errors数组）到stdout
6. 不执行VM，不交互询问

**静态辅助函数**：

- `ws2s(wchar_t*)`：宽字符转窄字符（仅用于ASCII token值）
- `wchar_to_utf8(wstring)`：UTF-16LE→UTF-8转换，正确编码中文错误信息
- `jsonEscape(string)`：JSON字符串转义
- `tokenKindName(int)`：38种token类型编号→字符串映射

## 3.3 前端模块（Frontend/）

前端通过Node.js桥接调用C++后端，浏览器内实现可视化编译与单步调试。词法/语法分析完全由后端完成，前端只负责UI渲染和虚拟机执行，避免重复实现。

### 3.3.1 Node.js桥接服务器（server.js）

**技术方案**：浏览器无法直接执行本地exe文件，需要HTTP中间层。server.js提供轻量级HTTP服务器，前端通过fetch POST源码到此服务器，服务器写临时文件后调用`l26compiler.exe --json`，将JSON结果返回前端。

**路由设计**：

| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/` | 读取`index.html`并返回，浏览器直接访问`localhost:8765`即可使用 |
| POST | `/compile` | 接收`{"source":"..."}`，写临时文件→调用编译器→返回JSON |
| OPTIONS | `/compile` | CORS预检，返回允许的Method和Header |

**POST /compile处理流程**：
1. 解析请求体JSON，提取source字段
2. 在系统临时目录创建`l26_<timestamp>.l26`文件，写入源码
3. 调用`execSync("l26compiler.exe <tmpFile> --json")`，超时10秒
4. 成功：返回`200` + JSON（tokens + pcode + errors）
5. 失败：返回`500` + 错误JSON
6. 删除临时文件

**关键设计**：server.js放在Frontend/目录下，编译器路径为`../Compiler/l26compiler.exe`，保持前后端目录分离。

### 3.3.2 可视化前端页面（index.html）

结构分三部分：CSS样式、HTML布局、JavaScript逻辑。

**界面布局**：

```
┌──────────────┬──────────────┬──────────────────┐
│   控制栏（编译+执行/仅编译/单步/示例/清空/状态）      │
├──────────────┼──────────────┼──────────────────┤
│   源代码      │   程序输出     │  内存栈(Frames)  │
├──────────────┴──────────────┼──────────────────┤
│   P-Code指令序列(50%)         │  Token列表(50%)  │
└─────────────────────────────┴──────────────────┘
```
**F码和OPR码对齐**：前端JS定义了与后端PCode.h完全一致的枚举：

- `F = {LIT:0, OPR:1, LOD:2, STO:3, CAL:4, INT:5, JMP:6, JPC:7}`
- `OPR`对象包含全部标准子码和集合扩展子码，数值与后端`OprCode`枚举一一对应

**JS虚拟机**：

- 数据结构：`s[2048]`整型栈、`setPool{}`集合池、`p/b/t`寄存器，与后端VM.h完全对齐
- `base(l, b)`：静态链寻址，沿`s[b]`向上l层
- `stepAsync()`：单步执行一条指令，返回true表示需要read输入
- `execOPR(ins)`：分发全部OPR子码，逻辑与后端VM.h一致
- `pushSet(set)`/`popSet()`：集合辅助存储，`setPool[t]`存Set对象
- `setStr(s)`：集合格式化输出（升序排列）

**编译流程（doCompile函数）**：

1. `fetch(BACKEND, {method:'POST', body:JSON.stringify({source})})` 发往后端
2. 解析JSON响应，提取tokens、pcode、errors
3. 先渲染Token表和P-Code表
4. 如有错误：显示在输出区，设置`compileHadErrors=true`，保存到`savedErrors`
5. 返回`compiledCode.length > 0`

**单步调试流程**：

1. 点击"开始单步"→`startStep()`→编译→初始化VM→清空输出区→显示"有编译错误，可单步查看"提示
2. 点击"单步执行"→`stepOnce()`→`vm.stepAsync()`执行一条指令→高亮当前PC行→更新内存栈帧→更新输出
3. 每步检查当前P-Code指令行号是否匹配编译错误行号，命中则在输出区显示错误（同一错误只显示一次）
4. read语句触发`promptInput()`弹窗，用户输入后继续执行
5. vm.halted时自动禁用单步按钮

**渲染函数**：

| 函数 | 功能 |
|------|------|
| `renderTokens(tokens)` | 遍历JSON token数组，按关键字/标识符/数字/分隔符/运算符分类着色 |
| `renderPCode(code)` | 遍历P-Code数组，显示地址/F名称(l a)/备注 |
| `highlightPCode(pc)` | 高亮指定PC行并滚动到可见区域 |
| `renderMemory(vm)` | 沿静态链向上遍历所有帧，从外到内渲染每帧卡片，显示SL/DL/RA联系单元和全部局部变量，集合变量展开为`{1,2,3}`，当前帧紫色高亮 |
| `renderOutput(output)` | 展示VM运行输出 |
| `updateStepInfo(vm)` | 更新PC/b/t寄存器值，显示栈顶最近6个元素 |

**错误展示策略**：

- 编译+执行模式：错误直接显示在输出区，阻止执行
- 单步模式：编译时不清空输出区，只显示淡黄提示；单步到对应行时展示错误；`shownErrors` Set确保每条错误只显示一次
- 后端返回的UTF-8中文错误消息正确渲染

**输入弹窗**：read语句触发Promise-based弹窗，用户输入整数后VM推送该值并继续执行。支持Enter键快速确认。

# 4 类P-Code规范与指令集设计

## 4.1 统一三元组格式
所有指令固定格式：`(地址) 助记符 l a`

## 4.2 八种基础指令

| 指令 | 格式 | 语义 |
|------|------|------|
| LIT | LIT 0 a | 常数a压栈 |
| OPR | OPR 0 x | 执行OPR子码x |
| LOD | LOD l a | 加载base(l,b)+a处变量压栈 |
| STO | STO l a | 弹出栈顶存入base(l,b)+a |
| CAL | CAL l a | 调用过程（L26暂不使用） |
| INT | INT 0 a | 栈顶指针上移a（a=局部变量数+3） |
| JMP | JMP 0 a | 无条件跳转 |
| JPC | JPC 0 a | 弹出栈顶，为0则跳转 |

## 4.3 OPR子码（与PL/0标准兼容 + 集合扩展）

**标准子码（0-20）：**
RET(0) / NEG(1) / ADD(2) / SUB(3) / MUL(4) / DIV(5) / ODD(6) /
EQ(8) / NE(9) / LT(10) / GE(11) / GT(12) / LE(13) /
WRITE(14) / WRITELN(15) / READ(16) /
AND(18) / OR(19) / NOT(20)

**集合扩展子码（201-211）：**
SET_MK(201)：弹cnt个整数构造集合（cnt存l字段）
SET_UNION(202)：弹两集合求并集
SET_INTER(203)：弹两集合求交集
SET_IN(204)：弹整数和集合测成员
SET_ISEMPTY(205)：测栈顶集合是否空
SET_ADD(206)：弹整数加入栈顶集合
SET_REMOVE(207)：弹整数从栈顶集合删除
SET_EQ(208)：测试两集合相等
WRITE_BOOL(209)：输出布尔值
WRITE_SET(210)：输出集合
SET_NE(211)：测试两集合不等

## 4.4 虚拟机栈帧布局

```
s[b+0] = SL（Static Link，外层基址）
s[b+1] = DL（Dynamic Link，调用方基址）
s[b+2] = RA（Return Address，返回地址）
s[b+3] = 局部变量0
s[b+4] = 局部变量1
...
```

Base(l,b)函数沿静态链向上l层寻址。

前端fetch POST源码到`localhost:8765/compile` → Node.js server写临时文件 → 调用`l26compiler.exe --json` → 返回JSON（tokens + pcode + errors）→ 前端渲染并执行

## 8.2 AI使用说明

开发过程中使用了 WorkBuddy 和 Claude Code 作为辅助工具。AI 的主要作用为代码生成优化和调试辅助——帮助快速生成框架性代码、排查语法错误、撰写文档草稿。所有 AI 输出的代码均经过我逐行审阅、理解、修改和验证，核心设计决策和语义逻辑由我独立完成。

#### 编译器核心模块：

整体架构和核心逻辑由我自己完成。AI 主要发挥了以下辅助作用：

- **L26.atg 文法文件**：文法规则本身由我独立设计完成。AI 辅助了 Coco/R ATG 格式语法的适配，比如将非标准 pragma 写法迁移到标准语法、解决编译时的格式报错。
- **PCode.h（P-Code 指令定义与代码生成器）**：PL/0 标准 8 条指令（LIT/OPR/LOD/STO/CAL/INT/JMP/JPC）16 个 OPR 子码和扩展的集合操作子码由我自己根据课件定义。AI 辅助了 C++ 结构体封装、跳转回填等的实现细节。
- **VM.h（虚拟机）**：指令执行循环和栈帧管理由我对标 PL/0 标准实现。AI 辅助了具体写法优化、边界条件检查等的补充。
- **SymTab.h（符号表）**：符号表结构、作用域入栈出栈、变量查找逻辑由我自己设计。AI 辅助了代码组织结构的优化和模板化封装。
- **main.cpp**：双模式架构设计、整体流程控制由我完成，AI 辅助了 `--json` 输出模式用于与前端对接的实现。

#### 前端可视化界面：

`Frontend/index.html` 和 `Frontend/server.js` 的整体代码由 AI 生成，但并非 AI"自主决定"写什么。具体流程是：

1. 由我决定前端需要三个核心功能：P-Code 指令列表展示、运行时栈/内存可视化、逐步执行调试；
2. 我指定技术方案：纯 HTML + JS 实现前端虚拟机、Node.js 做桥接服务调用后端 exe；
3. 我要求前端 JS 虚拟机的指令执行逻辑必须与后端 C++ VM.h 逐条对齐，确保前端展示的运行状态与后端一致；
4. AI 生成代码后，我逐段审阅、测试，修正了集合类型在前端渲染时的显示 bug，调整了界面布局使其更直观。

#### 测试用例：

- 我自己先编写了几个基础测试用例；
- 在这些用例通过后，我让 AI 基于我的用例风格和 L26 语言规范，补充了更完善的测试用例，包括边界条件（空集合、嵌套作用域变量遮蔽）、组合场景（布尔+集合混合表达式）、各类语义错误检测（未声明变量、类型不匹配、重复声明）等。

#### 使用总结

AI 主要承担了前端展示层和文档辅助工作。编译器的核心技术部分均由人工主导完成。AI 在调试冲突、发现 token 编号错误、生成测试边界用例等方面起到了有效的辅助作用，但所有设计决策均经我确认。这次在编译原理大作业的开发过程使用ai，让我对"怎么和 AI 协作"有了不少体会：

1. **AI 能加速实现，但不能替代理解。** 最典型的是 P-Code 虚拟机。AI 可以五分钟写出一个能跑的 VM 框架，但如果我不理解 PL/0 的栈帧布局和静态链机制，出了 bug 根本无从查起。AI 帮我省掉的是敲键盘的时间，不是啃课件的时间。
2. **需求描述越精确，AI 产出越靠谱。** 前期我给 AI 的指令比较模糊（比如"帮我写个前端"），出来的东西偏离很大。后来我学会了先把功能拆解清楚、指定技术路线（纯 HTML + JS、不需框架）、甚至给出数据结构协议（前后端 JSON 字段名和格式），AI 的代码质量立刻提升很多。
3. **AI 会犯错。** 有几次 AI 生成的代码在编译时报错，我丢回去让它修，它又生成一段同样有问题的代码，来回好几次。最后是我自己看报错信息、对比 ExpressionProject 模板修好的。
4. **对"AI 辅助开发"有了边界感。** 哪些事可以放心交给 AI（框架性代码、重复性语义动作、文档初稿），哪些事必须自己把控（文法设计、核心算法、类型系统规则、最终验证），这次实践帮我逐渐建立了一套判断标准。这对以后在更大的工程里使用 AI 会很有帮助。
