# L26编译器：栈、内存管理与集合相等实现详解

---

# 一、栈式虚拟机架构

## 1.1 整体模型

L26编译器采用**标准PL/0栈式虚拟机**模型。这不是"用栈实现某个算法"，而是**整台虚拟机本身就是一台栈机器**——所有计算、存储、函数调用全部通过一个整型数组栈完成。

### 四个核心寄存器

| 寄存器 | 含义 | 初始值 | 作用 |
|--------|------|--------|------|
| `s[2048]` | 操作数栈 | 全0 | 唯一的数据存储空间 |
| `p` | Program Counter | 0 | 当前待执行指令在code数组中的下标 |
| `b` | Base Register | 0 | 当前活动记录（栈帧）在栈中的起始位置 |
| `t` | Top-of-stack | -1 | 栈顶指针，始终指向最后压入的元素 |

### 为什么用栈？

PL/0（以及所有P-Code变体）选择栈式架构的原因：
1. **表达式求值天然适合栈**：`1 + 2 * 3` → 压1、压2、压3、乘、加，无需寄存器分配
2. **嵌套作用域天然适合栈**：进入块→栈顶增长，离开块→栈顶收缩，无需malloc/free
3. **指令集极简**：零地址指令（操作数隐含在栈顶），每条指令只需f/l/a三个字段

## 1.2 栈的全生命周期

以程序 `{ int x; int y; x = 3; y = x + 1; write y; }` 为例，逐条追踪栈的变化。

### 初始状态
```
p=0, b=0, t=-1
s: [0, 0, 0, 0, 0, 0, 0, ...]
```

### 指令(0)：INT 0 5  （分配活动记录：2个变量 + 3个联系单元）
```
执行: t += 5  →  t = 4
p=1, b=0, t=4
s: [?, ?, ?, ?, ?, 0, 0, ...]
       ↑b=0          ↑t=4
```
`INT 0 5` 在栈顶分配5个槽。为什么是5？因为声明了2个变量(x,y)，活动记录需要3+2=5个槽。变量数由编译器在编译时统计（`Decls`产生式计数），回填到INT指令的a字段。

INT指令**不初始化栈内容**，只是移动t指针。后续STO指令才会写入实际值。

### 指令(1)：LIT 0 3  （压入常数3）
```
执行: t++; s[t] = 3  →  t = 5
p=2, b=0, t=5
s: [?, ?, ?, ?, ?, 3, 0, ...]
                ↑t=5
```

### 指令(2)：STO 0 3  （弹出栈顶，存入变量x）
```
执行: s[Base(0,0) + 3] = s[5]; t--
     s[0 + 3] = 3  →  s[3] = 3; t = 4
p=3, b=0, t=4
s: [?, ?, ?, 3, ?, 0, ...]
          ↑x     ↑t=4
```
`STO l a` 的寻址过程：
1. `Base(l, b)` 沿静态链向上l层（l=0表示当前层），得到目标基址
2. 偏移 `a = 3` 是变量在帧内的位置。为什么是3？因为前3个槽固定为SL/DL/RA，局部变量从偏移3开始
3. 栈顶值3被写入 s[3]，然后t减1（弹出）

### 指令(3)：LOD 0 3  （加载变量x的值压栈）
```
执行: t++; s[t] = s[Base(0,0) + 3]  →  s[5] = s[3] = 3; t = 5
p=4, b=0, t=5
```

### 指令(4)：LIT 0 1
```
执行: t++; s[t] = 1  →  t = 6
s: [?, ?, ?, 3, ?, 3, 1, ...]
                     ↑t=6
```

### 指令(5)：OPR 0 2  （ADD：弹出两个值，压入和）
```
执行: t--; s[t] = s[t] + s[t+1]
     t=5; s[5] = 3 + 1 = 4
p=6, b=0, t=5
s: [?, ?, ?, 3, ?, 4, 0, ...]
                  ↑t=5
```
`OPR 0 2` 的子码a=2对应ADD运算：弹出栈顶两个元素，相加后压回。**注意**：第一个弹出的在右边（s[t+1]），第二个在左边（s[t]），即 `左操作数 OP 右操作数`。

### 指令(6)：STO 0 4  （弹出栈顶，存入变量y）
```
执行: s[Base(0,0) + 4] = s[5]  →  s[4] = 4; t = 4
p=7, b=0, t=4
s: [?, ?, ?, 3, 4, ?, ...]
          ↑x  ↑y  ↑t=4
```

### 指令(7)：LOD 0 4 + 指令(8)：OPR 0 14  （加载y + 写整数）
```
执行: t++; s[t] = s[4] = 4  →  t=5
      OPR 0 14 (WRITE) → 输出"4", t-- → t=4
p=9, b=0, t=4
输出: 4
```

### 指令(9)：OPR 0 0  （RET：销毁活动记录）
```
执行: t = b - 1 = -1
      p = s[b+2] = s[2]  →  p = 0 (halted)
      b = s[b+1] = s[1]  →  b = 0
p=0 (halted), b=0, t=-1
```
RET指令将栈恢复到进入块之前的状态：t回到b-1，p和b从RA和DL恢复。主程序的RA=0，所以p=0触发halted。

# 二、内存管理机制

## 2.1 活动记录（Activation Record / Stack Frame）

每进入一个`{}`块，编译器发出`INT 0 (n+3)`指令在栈顶开辟活动记录。每个活动记录的结构固定：

```
      高地址（栈增长方向 ↓）
      ┌──────────────────┐
      │   局部变量 n-1   │  ← s[b+2+n]
      │      ...         │
      │   局部变量 0     │  ← s[b+3]
      ├──────────────────┤
      │   RA (返回地址)  │  ← s[b+2]  函数返回后p应该跳到哪里
      ├──────────────────┤
      │   DL (动态链)    │  ← s[b+1]  调用者的b值
      ├──────────────────┤
      │   SL (静态链)    │  ← s[b+0]  外层作用域的b值（词法嵌套）
      └──────────────────┘  ← b (基址寄存器指向这里)
      低地址
```

**三个联系单元的作用**：

| 单元 | 全称 | 作用 | 被谁使用 |
|------|------|------|----------|
| SL | Static Link | 指向**词法上**外层作用域的b值 | LOD/STO指令通过Base()访问跨层变量 |
| DL | Dynamic Link | 指向**调用链上**调用者的b值 | RET指令恢复调用者的b |
| RA | Return Address | 过程返回后p应该设置为多少 | RET指令恢复p |

在L26中，由于没有函数定义，SL和DL始终相同（都指向外层block的b），RET指令使p=0触发程序终止。

## 2.2 嵌套作用域的栈帧布局

以PDF示例2为例：
```l26
{
    int x;      // 外层：x是int，偏移3
    set s;      //       s是set，偏移4
    x = 10;
    s = {1, 2};
    {            // ← 进入内层块
        set x;   // 内层：x是set，遮蔽外层x，偏移3
        x = {5, 6};
        add x 7;
        write x;
        write s; // ← 访问外层的s
    }            // ← 离开内层块，恢复外层x
    write x;     // ← x又变回int 10
}
```

**编译时分配**：

外层块声明了两个变量：
```
SymTab状态（进入外层块后）:
  curLevel = 0
  scopes[0] = {
    "x" → {type:"int",  level:0, offset:0}
    "s" → {type:"set",  level:0, offset:1}
  }
```

外层块INT指令的参数 = 2个局部变量 + 3 = **5**。

内层块声明了一个变量：
```
SymTab状态（进入内层块后）:
  curLevel = 1
  scopes[0] = {"x"→{type:"int", level:0, offset:0}, "s"→{type:"set", level:0, offset:1}}
  scopes[1] = {"x"→{type:"set", level:1, offset:0}}
```
内层块INT指令的参数 = 1个局部变量 + 3 = **4**。

**运行时栈的变化**：

```
进入外层块后：                   进入内层块后：                  离开内层块后：
s[0]=SL(0)   ← b=0             s[0]=SL(0)                     s[0]=SL(0)   ← b=0
s[1]=DL(0)                      s[1]=DL(0)                     s[1]=DL(0)
s[2]=RA(0)                      s[2]=RA(0)                     s[2]=RA(0)
s[3]=10      (int x)            s[3]=10      (int x)           s[3]=10      (int x)
s[4]=idx_s   (set s)            s[4]=idx_s   (set s)           s[4]=idx_s   (set s)
                    ← t=4       s[5]=SL(0)   ← b=5             s[5]=...     ← t=4
                                s[6]=DL(0)                     (内层帧已销毁)
                                s[7]=RA(0)
                                s[8]=idx_x'  (set x, 遮蔽外层)
                                                    ← t=8
```

**变量寻址**：

访问外层的s（在内层块中`write s`）：
```
Lookup("s") → {type:"set", level:0, offset:1}
LevelDiff = curLevel(1) - defLevel(0) = 1
LOD 1 (1+3) → LOD 1 4
VM: Base(1, b=5) → s[5]=0 → 返回0
    s[t+1] = s[0 + 4] = s[4] = idx_s  ✓
```
l=1表示"向上跨1层"，Base()从内层b=5出发，取`s[5]`（内层帧的SL=0），返回0。然后在基址0上加偏移4，访问s[4]——这正是外层变量s的存储位置。

**变量遮蔽**：内层的`set x`和外层的`int x`在符号表的不同层。Lookup从内向外查找，先找到内层的`set x`。离开内层后，`scopes[1]`被弹出，Lookup只能找到外层的`int x`。

## 2.3 符号表：编译时的内存管理

编译时，符号表（SymTab.h）通过**栈式数据结构**管理变量声明与作用域：

```
EnterScope() → 压入新层
  scopes: [{...}] → [{...}, {...}]

Declare("x", "int") → 在当前层分配偏移
  scopes[curLevel]["x"] = {offset: offsetStack[curLevel]++}
  offsetStack: [0] → [1]

Declare("y", "bool") → 
  scopes[curLevel]["y"] = {offset: 1}
  offsetStack: [1] → [2]

Lookup("x") → 从顶向下搜索
  scopes[1]["x"]? → 没找到
  scopes[0]["x"]? → 找到了！返回 {type:"int", level:0, offset:0}

LeaveScope() → 弹出顶层
  scopes: [{...}, {...}] → [{...}]
```

变量偏移+3的约定贯穿始终：符号表分配的偏移从0开始，代码生成时统一加3。这样偏移0→地址3、偏移1→地址4，恰好跳过SL/DL/RA三个联系单元。

## 2.4 集合的内存管理

集合不直接存储在整型栈`s[]`中。`s[]`中只存一个整数index，作为键去`g_setPool`中查找实际的`std::set<int>`对象。

```
s[4] = 4  ──────────→  g_setPool[4] = {1, 2, 3}
                         (std::set<int> 对象)
```

**为什么这样设计？**
- PL/0的栈原本只支持整数。如果直接把`std::set<int>`塞进`s[]`，会破坏整个栈的整数语义
- 用一个平行数组存储集合，整数栈只存index，不改变原有栈操作逻辑
- LOD/STO指令无需感知集合类型：LOD时检查`g_setPool[addr]`是否存在，存在则复制集合；STO时检查栈顶是否有集合映射，有则写入`g_setPool`

**集合字面量的栈操作**（`s = {1, 2, 3}`）：

```
LIT 0 1    → t=5, s[5]=1
LIT 0 2    → t=6, s[6]=2
LIT 0 3    → t=7, s[7]=3
OPR 3 201  → 弹3个元素 {1,2,3} → t=5, s[5]=5, g_setPool[5]={1,2,3}
              ↑ l=3(cnt)  ↑ a=201(SET_MK)
STO 0 4    → s[4] = s[5] = 5, t=4
              g_setPool[4] = g_setPool[5] = {1,2,3}
```

SET_MK指令将cnt存在l字段（`Emit(F_OPR, cnt, OPR_SET_MK)`），VM从l字段读取需要弹出的元素个数。注意：SET_MK不消耗额外栈空间——弹出cnt个整数后压入1个集合引用，净栈变化是t减少cnt-1。

# 三、集合相等/不等的完整实现链

这是项目的核心加分项（+10分）。实现跨越了三个层次：语法分析识别、代码生成选择、虚拟机执行。

## 3.1 第一步：编译器识别操作数类型

在L26.atg的Expr产生式中，当解析到关系运算符时，编译器追踪左侧表达式的类型：

```cpp
AExpr<typ>            // ← typ被设为左侧表达式的类型（"int"或"set"）
[ ( "==" | "!=" )     // 关系运算符
  AExpr<dummy>        // 右侧表达式
  (. prevTyp = typ; typ = "bool";
     if (prevTyp == "set") {          // ← 关键判断
         if (relOp == 5) pcode.EmitSetEQ();     // == → SET_EQ
         else if (relOp == 6) pcode.EmitSetNE(); // != → SET_NE
         else SemError(L"集合只支持 == 与 !=");   // <、<=等报错
     } else {
         pcode.EmitRelOp(relOp);      // int类型走标准关系运算
     }
  .)
]
```

同样逻辑在BFactor产生式中重复（处理`aexpr relop aexpr`作为布尔因子的情况），两份代码保持一致。

**编译时类型追踪流程**：
```
源码:  r = (a == b);
      其中 a: set, b: set

词法分析 → Token流: ident("r") = ( ident("a") == ident("b") ) ;

语法分析 → Expr产生式:
  1. AExpr<typ> → 解析 "a" → Lookup("a")→{type:"set"} → typ="set"
     发射 LOD 0 (offset_a+3)
  2. "==" → relOp = 5
  3. AExpr<dummy> → 解析 "b"
     发射 LOD 0 (offset_b+3)
  4. 类型判断: prevTyp("set") == "set" → 走集合分支
     relOp=5 → EmitSetEQ()
     发射 OPR 0 208 (SET_EQ)
  5. typ = "bool" (表达式整体返回bool类型)
  6. AssignStmt: e->typName("bool") == "bool" ✓
     发射 STO l (offset_r+3)
```

## 3.2 第二步：代码生成

PCode.h中集合相等的发射函数：

```cpp
void EmitSetEQ() { Emit(F_OPR, 0, OPR_SET_EQ); }
// OPR_SET_EQ = 208
```

生成的P-Code指令为 `OPR 0 208`，即f=OPR(1), l=0, a=SET_EQ(208)。

集合不等的发射函数：
```cpp
void EmitSetNE() { Emit(F_OPR, 0, OPR_SET_NE); }
// OPR_SET_NE = 211
```

完整的P-Code序列（`r = (a == b);`，a和b均为set）：
```
(k+0) LOD 0 3     ; 加载a（栈上存set pool index）
(k+1) LOD 0 4     ; 加载b
(k+2) OPR 0 208   ; SET_EQ：弹出两集合，比较后压入0/1
(k+3) STO 0 5     ; 存入r
```

## 3.3 第三步：虚拟机执行

VM.h中SET_EQ的执行：

```cpp
case OPR_SET_EQ: {
    int eq = (g_setPool[t-1] == g_setPool[t]) ? 1 : 0;
    t--;
    s[t] = eq;
    break;
}
```

**逐步骤分解**：

1. **栈状态**（执行前）：
   ```
   t=5
   s[4] = 4  →  g_setPool[4] = {1, 2, 3}  (集合a)
   s[5] = 5  →  g_setPool[5] = {1, 2, 3}  (集合b)
   ```

2. **比较**：`g_setPool[t-1] == g_setPool[t]` 即 `g_setPool[4] == g_setPool[5]`
   - 这是C++的 `std::set<int>::operator==`
   - 标准规定：两个set相等当且仅当**元素完全相同**（元素值相同、元素数量相同），**与元素插入顺序无关**
   - `{1, 2, 3} == {1, 2, 3}` → **true**
   - `{1, 2, 3} == {3, 2, 1}` → **true**（顺序无关）
   - `{1, 2, 3} == {1, 2}` → **false**（元素数量不同）
   - `{1, 2, 3} == {1, 2, 4}` → **false**（元素值不同）

3. **压入结果**：
   ```
   t减1: t = 4
   s[4] = 1 (true)
   ```
   栈上两个set引用被弹出，替换为整数0或1（标准PL/0布尔表示）。

SET_NE的实现完全对称：
```cpp
case OPR_SET_NE: {
    int ne = (g_setPool[t-1] != g_setPool[t]) ? 1 : 0;
    t--;
    s[t] = ne;
    break;
}
```
利用`std::set::operator!=`，逻辑为SET_EQ的取反。

## 3.4 为什么std::set::operator==是正确选择？

`std::set`底层是**红黑树**，`operator==`的比较算法是：
1. 先比较两棵树的`size()`——不同则直接返回false
2. 用 `std::equal(first1, last1, first2)` 逐元素比较——set内部有序，两个有序序列的逐元素比较即等同于集合相等

**时间复杂度**：O(n)，n为集合元素个数。

对于集合上限200的限制，单次比较最多200次整数比较，性能完全可以接受。

## 3.5 前端JS VM中的对应实现

前端index.html中的JS VM做了相同实现：

```javascript
const F = { LIT:0, OPR:1, LOD:2, STO:3, CAL:4, INT:5, JMP:6, JPC:7 };
const OPR = {
    // ...
    SET_EQ: 208,
    SET_NE: 211,
    // ...
};

// VM.execOPR中:
case OPR.SET_EQ: {
    const sb = this.popSet(), sa = this.popSet();
    this.push(sa.size === sb.size && [...sa].every(x => sb.has(x)) ? 1 : 0);
    break;
}
case OPR.SET_NE: {
    const sb = this.popSet(), sa = this.popSet();
    this.push(sa.size !== sb.size || ![...sa].every(x => sb.has(x)) ? 1 : 0);
    break;
}
```

JS没有原生的集合相等运算符，所以手动实现：先比较size，再检查sa的每个元素是否都在sb中。逻辑等价于C++的`std::set::operator==`。

前后端F码和OPR子码数值完全一致，确保同一份P-Code在C++ VM和JS VM中执行结果相同。

# 四、总结

| 概念 | 实现位置 | 核心机制 |
|------|----------|----------|
| 运行时栈 | VM.h s[2048] | 整型数组 + p/b/t三寄存器，所有运算通过栈完成 |
| 活动记录 | VM.h INT/RET指令 | 进块分配n+3槽，退块回收，SL/DL/RA三联系单元 |
| 跨层变量访问 | VM.h Base(l, b) | 沿SL链向上l层，课件标准静态链寻址算法 |
| 编译时作用域 | SymTab.h | vector<map>栈式多级符号表，EnterScope/LeaveScope |
| 变量偏移约定 | SymTab.h + ATG | 符号表偏移从0起，代码生成+3跳过联系单元 |
| 集合运行时存储 | VM.h g_setPool | 平行数组，栈中存index，不破坏整型栈语义 |
| 集合构造 | PCode.h EmitSetMake | cnt存l字段，VM弹出cnt个整数构造set |
| 集合相等检测 | PCode.h EmitSetEQ + VM.h SET_EQ | std::set::operator==，O(n)逐元素比较 |
| 集合不等检测 | PCode.h EmitSetNE + VM.h SET_NE | std::set::operator!=，对称实现 |
| 前后端一致性 | index.html JS VM | F码/OPR码数值完全对齐，相同P-Code相同结果 |
